# ADR-001: Desacoplamiento de Mensajería mediante Redis, BullMQ y Worker Independiente## Estado**Aceptado**
## ContextoEl sistema requiere procesar tareas asincrónicas de mensajería (inicialmente el envío de correos electrónicos transaccionales y, a futuro, notificaciones push). El backend principal del proyecto está desarrollado sobre el framework **Next.js**. 

Ejecutar tareas de larga duración o integraciones con APIs de terceros directamente en el ciclo de vida de las peticiones HTTP de Next.js degrada la experiencia del usuario (UX) debido a la latencia. Además, el entorno de despliegue objetivo para el frontend está orientado a arquitecturas Serverless (e.g., Vercel), las cuales imponen límites estrictos de tiempo de ejecución (*timeouts*) e implementan apagados automáticos (*scale-to-zero / cold starts*) ante la inactividad, lo que imposibilita mantener conexiones persistentes para procesos en segundo plano (*background workers*).
## DecisiónSe decide separar la arquitectura en **tres componentes desacoplados**:1. **Emisor (Next.js):** Genera tickets de eventos mínimos e instantáneos.2. **Broker de Mensajería (Upstash Redis):** Almacena de forma segura y en memoria la cola de tareas pendientes.3. **Consumidor (Worker Node.js/TypeScript Dedicado):** Un microservicio simplificado de ciclo de vida largo (ejecutándose 24/7 en una plataforma PaaS como Render o Railway) que consume la cola utilizando la librería **BullMQ** y despacha las peticiones hacia el proveedor de infraestructura final (inicialmente **Resend** para emails).
---### Tradeoffs y Justificación del Stack Seleccionado#### 1. Separación de Next.js vs. Worker Monolítico* **Decisión:** El Worker se extrae a un script independiente de Node.js/TypeScript.* **Justificación:** Next.js se libera de la carga computacional de procesamiento pesado. Al aislarlo, el frontend puede ser hosteado en plataformas Serverless eficientes sin que el algoritmo de apagado del host destruya o pause la ejecución de la cola de mensajes pendientes. El Worker mantiene una conexión TCP fija y estable con Redis las 24 horas del día.
#### 2. Motor de Colas: BullMQ (Redis) vs. Apache Kafka* **Decisión:** Se selecciona BullMQ sobre infraestructura Redis.* **Justificación:** Kafka está diseñado para la ingesta masiva de telemetría y sincronización compleja de microservicios corporativos; su TCO (Costo Total de Propiedad) y complejidad operativa destruirían el *time-to-market* en esta fase del proyecto. BullMQ ofrece abstracciones nativas en TypeScript para reintentos automáticos, control de flujo (*rate limiting*) y retrasos de tareas con un consumo menor a 100MB de RAM.* **Proveedor (Upstash Redis):** Se elige por su capa gratuita Serverless orientada a desarrollo y su compatibilidad con el protocolo TCP nativo de Redis, evitando costos de infraestructura ociosa en etapas iniciales.
#### 3. Hosting del Worker: PaaS (Render / Railway) vs. AWS (EC2/ECS) vs. VPS Limpio* **Decisión:** Se opta por plataformas PaaS con capas gratuitas/créditos de inicio.* **Justificación:** Reduce a cero la carga de operaciones (DevOps). Automatizan el pipeline de compilación, rotación de logs y reinicios automáticos ante fallas. Evita la sobre-ingeniería de configurar VPCs o políticas de IAM en AWS, y elimina el riesgo de mantenimiento de un VPS puro (donde el desarrollador debe parchar el sistema operativo y gestionar demonios).
#### 4. Proveedor de Email: Resend vs. Amazon SES* **Decisión:** Se adopta Resend para la fase inicial.
* **Justificación:** Máxima velocidad de desarrollo gracias a su API limpia y su integración nativa con `React Email`, permitiendo diseñar plantillas utilizando TypeScript y React. Su capa gratuita cubre las necesidades del MVP (3,000 correos/mes). 
---## Fortalezas ante las Funcionalidades Requeridas (Emails y Push)* **Resiliencia ante Fallas Externas:** Si el proveedor de emails (Resend) o el de notificaciones push (e.g., Firebase) sufre una caída, los usuarios no experimentarán errores en la aplicación web. Los eventos se acumulan de forma segura en la memoria de Redis y el Worker los procesará automáticamente con retraso exponencial (*backoff*) cuando los servicios externos se recuperen.* **Evolución y Escalabilidad Futura:** 
  * **Escala Horizontal:** Si el volumen de notificaciones push o correos satura el servidor, se pueden levantar múltiples instancias idénticas del Worker en paralelo sin modificar una sola línea de código; BullMQ distribuirá la carga equitativamente.
  * **Migración de Infraestructura (Cero Vendor Lock-in):** Si el costo de Resend se vuelve prohibitivo a gran escala, la arquitectura modular permite reemplazar su SDK por el de Amazon SES exclusivamente dentro del archivo del Worker, sin afectar a Next.js ni a la base de datos de Redis.* **Control de Flujo (Anti-Spam / Rate Limiting):** BullMQ permite configurar de forma nativa que el Worker procese, por ejemplo, un máximo de 5 tareas por segundo. Esto protege la reputación del dominio frente a los servidores de correo (Gmail/Outlook) y evita violar las cuotas estrictas de las capas gratuitas de los proveedores.
## Consecuencias* Se deben mantener dos repositorios (o un directorio separado dedicado) con configuraciones de `package.json` independientes para aislar las dependencias del Worker de las de la aplicación web.
* Es imperativo asegurar que la cadena de conexión de Redis (`REDIS_HOST`) esté correctamente configurada como variable de entorno compartida y segura en ambas plataformas de hospedaje.
